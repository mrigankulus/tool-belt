Events & Notifications

The notifications service allows you to notify users of events that occur in your application, and records a history of those events to show the user a record of activity for your resources. In addition to saving you time in design and development, this service encourages a unified user experience and assembles this data into an ecosystem that can be leveraged for future applications.


Create a Resource Type & Event Types

Your resource type will represent the domain of your events (e.g. "Quote" or "Order").
Your event types are the definitions or templates of how your events will look in emails, the notifications drop down, and the resource event stream (e.g. "Quote Won" or "Order Shipped"). Use the tool belt to create resource types and event types.

Sending Notifications

To send a notification, you'll publish an event to RabbitMQ using the exchange defined in your event type's settings. The notifications service has set up dynamic bindings to your exchange. It will pickup your events and process them accordingly. If sending RabbitMQ messages is not an option, the events-api allows you to publish RabbitMQ events with an HTTP POST.

Users can subscribe to your event types to receive these notifications from the notifications application. You can also subscribe users from your resource and/or event types in the tool belt. For more advanced usage, use the subscriptions api to subscribe users.


A Few Important Notes

  • If you're firing events from your API, you'll need to be connected to DEV (not SAND) for the process to work locally.
  • Emails triggered by events in DEV will go to your DEV email account. If you don't already have access to your DEV email account, you'll need to submit a request in service now.
  • Please don't be shy to create an issue with any questions, concerns or feedback about the notifications service.
